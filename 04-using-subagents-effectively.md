# 第四章：Using subagents effectively -- 高效使用 Subagent

> **课程章节**: 4/4 | **视频时长**: 4:44
> **视频链接**: https://www.youtube.com/watch?v=n5LoKZ8Oa-A

---

你已经知道如何创建 subagent 并且设计好它们了。现在的问题是：它们什么时候真正有帮助，什么时候反而碍事？区别归结为一件事——**中间过程的工作对你的主线程是否重要**。

> **[Claude 注释]** 这是整门课程最核心的判断准则，值得反复品味。它本质上是一个 **信息可见性 vs 效率** 的权衡：
>
> - 需要看到过程 → 留在主线程
> - 只需要结果 → 委派给 subagent
>
> 这和日常管理中的"微管理 vs 委派"完全一样。

---

## 什么时候 Subagent 大放异彩

Subagent 在**探索与执行分离**的情况下效果最好。如果任务中的每一步都依赖于前一步发现的内容，你希望那些工作留在主线程中。但如果你只需要一个答案，不关心寻找答案的过程，就委派出去。

Subagent 擅长的任务特征：

- 你需要的是**结果**，而不是找到结果过程的逐步回放
- 探索性工作会**弄乱主线程的 context**
- 任务受益于一个**全新的视角**或一个**自定义的 system prompt**

---

## 研究任务

研究是 subagent 的经典用例。考虑在一个不熟悉的代码库中调查身份验证是如何工作的。你的主线程需要知道 JWT *在哪里*被验证，但不需要看到沿途搜索的每一个文件。

一个研究 subagent 可以读取几十个文件，追踪函数调用链，探索不同的代码路径。所有这些探索都留在 subagent 的 context 中。你的主线程收到一个干净的摘要，比如：

```
JWT validation happens in middleware/auth.js line 42, called from
the Express router in route/api.js
```

Subagent 做了繁重的工作。你的主线程得到了它继续前进所需的确切信息。

> **[Claude 注释]** 研究任务是 subagent 的"甜蜜点"（sweet spot），因为它完美符合三个条件：
> 1. **过程不重要**——你不关心搜索了哪 20 个文件，你只要答案
> 2. **过程很"吃" context**——每个文件读取都消耗大量 token
> 3. **结果可以被简洁表达**——一两句话就够了
>
> 其实 Claude Code 的内置 **Explore** subagent 就是专门为这种场景设计的。

## Code Review

当代码被呈现为由**别人编写**时，Claude 审查代码的效果更好。如果你在主线程中经过多轮对话构建了一个功能，让同一个线程来审查它通常会产生薄弱的反馈。Claude 参与了创建过程，所以它很难用全新的眼光来看待这段代码。

一个 reviewer subagent 在独立的 context 中看到这些变更。它运行 `git diff`，读取修改后的文件，并应用它专门的审查标准——而没有代码是如何被编写的历史记录。这种分离还让你可以在 subagent 的 system prompt 中编码项目特定的审查标准，确保团队之间审查标准的一致性。

> **[Claude 注释]** 这是一个非常深刻的洞察，值得展开：
>
> **为什么 Claude 很难审查自己写的代码？** 这和人类程序员的心理类似——当你花了 2 小时写一段代码后，你很难客观地审视它，因为你的大脑已经"认同"了这个实现。在 LLM 的世界里，这种"认同"体现在 context window 中——主线程的 context 里充满了"为什么选择这个方案"的推理过程，这些推理过程会偏置（bias）Claude 对代码的评价。
>
> **Subagent 打破了这种偏置**。因为 reviewer subagent 有一个全新的 context，它不知道代码是"经过深思熟虑后写出的"还是"随手写的"。它只看到代码本身——就像一个新加入团队的同事做 code review 一样。
>
> **最佳实践**：如果你用 Claude Code 写了一个复杂功能，在最后总是用一个 code review subagent 来审查。这相当于给你的代码增加了一层"第二双眼睛"的检查。

## 自定义 System Prompt

Claude Code 的默认 system prompt 强调简洁的、以代码为中心的响应。这对编码来说很棒，但不适合所有场景。

以下两个案例中，自定义 system prompt 让 subagent 比主线程做得更好：

- **Copywriting subagent（文案 subagent）**——给它关于语调、受众和风格的指令。Claude Code 的默认 prompt 倾向于简洁的技术写作，这对落地页或邮件营销来说真的不是你想要的。一个 copywriting subagent 可以拥有关于语音和结构的完全不同的指令。
- **Styling subagent（样式 subagent）**——将它指向你的设计系统文件。当 subagent 运行时，这些文件会自动加载到它的 context 中，所以它在开始写任何 CSS 之前就知道你的颜色变量、间距约定和组件模式。

> **[Claude 注释]** 这揭示了 subagent 的一个更深层的价值：**它不仅仅是"任务委派"，它还是"人格切换"**。
>
> 主线程的 Claude 被一个通用的"编程助手"system prompt 所定义。但通过 subagent，你可以创建完全不同的"人格"：
> - 一个"严格的安全审计员"
> - 一个"友好的文案写手"
> - 一个"精通你设计系统的 CSS 专家"
>
> 每个 subagent 都可以有完全不同的行为方式、关注点和输出风格——因为它们各自有自己的 system prompt。
>
> **类比**：主线程是一个"全栈工程师"，subagent 们是各个领域的"专家顾问"。你不会让全栈工程师用写代码的风格来写营销文案——同样，你不应该让默认的 Claude Code 来做需要完全不同"人格"的任务。

## 什么时候 Subagent 适得其反

启动 subagent 的开销——失去对其工作的可见性并将发现压缩成摘要——只有在 subagent 做了主线程**做不到**的事情时才值得。有三种常见的反模式需要注意。

### 反模式一：专家声明（Expert Claims）

声称拥有专业知识的 subagent 很少有帮助。像"you are a Python expert"或"you are a Kubernetes specialist"这样的 prompt 没有任何附加价值，因为 Claude 已经拥有这些知识。一个所谓的"专家 subagent"没有任何东西是你的主线程不能直接做到的。

> **[Claude 注释]** 这是一个极其常见的误区。很多人的第一反应是创建一堆"专家"subagent：Python 专家、React 专家、数据库专家等。但这是无效的，因为：
>
> 1. **Claude 已经具备这些知识**——它不需要一个"专家人设"来写好 Python 代码
> 2. **subagent 的开销不为零**——启动 subagent 有延迟，结果要被压缩成摘要，这都是成本
> 3. **没有结构性优势**——"专家 subagent"既没有特殊的 system prompt 指令（仅仅是"you are an expert"），也没有工具限制，也没有独特的输出格式
>
> **判断标准**：如果你的 subagent 的 system prompt 唯一的"特殊之处"就是声称它是某个领域的专家，那这个 subagent 就不值得创建。subagent 的价值来自于**结构性差异**（不同的工具、不同的输出格式、不同的行为指令），而不是来自"专家人设"。

### 反模式二：顺序流水线（Sequential Pipelines）

顺序 subagent 流水线会产生问题。考虑一个三个 agent 的流程：一个重现 bug，一个调试它，一个修复它。流水线在任务**真正独立**时才有效。当每一步都依赖于前一步的发现时——而 bug 修复几乎总是这样——它们就失败了。信息在 agent 之间的交接中丢失。

> **[Claude 注释]** 这是"分工"思维的一个典型陷阱。在人类团队中，我们习惯于把工作分成多个步骤交给不同的人。但 subagent 之间的"交接"比人类团队的交接**损失更大**，因为：
>
> 1. **每个 subagent 的 context 都会被丢弃**——Agent A 发现的所有细节都被压缩成一段摘要传给 Agent B
> 2. **摘要必然会丢失信息**——"我在 debug 过程中注意到 X 模块有一个微妙的竞态条件"这样的细节可能在摘要中被省略
> 3. **链条越长，信息损失越大**——就像传话游戏（telephone game），每一次转述都会丢失一些信息
>
> **什么时候流水线可以工作？** 当步骤之间**完全独立**时。比如"同时研究三个不相关的模块"——这是并行 subagent，不是顺序流水线，效果很好。

### 反模式三：测试运行器（Test Runners）

测试运行器 subagent 往往会隐藏你需要的信息。当测试失败时，你需要**完整的输出**来诊断问题。一个返回"tests failed"的 subagent 迫使你创建额外的调试脚本来获取在直接输出中本来就可见的细节。**测试表明，test runner 模式在所有配置中表现最差。**

> **[Claude 注释]** 这一点可能反直觉，但它非常有道理。让我解释为什么：
>
> 测试输出往往包含**你需要直接看到的诊断信息**：
> - 哪行代码断言失败了
> - 实际值 vs 期望值是什么
> - 完整的错误堆栈追踪
>
> 如果这些信息被 subagent 消化后变成一句"3 tests failed in auth module"，你失去了所有诊断线索。然后你不得不重新跑一遍测试来看完整输出——这完全违背了使用 subagent 的目的。
>
> **经验法则**：如果一个任务的输出本身就是你需要的"原始数据"（而不是需要被分析后提炼的信息），那就不要用 subagent。测试输出、编译错误、运行时日志——这些通常都属于"原始数据"。

## 决策规则

当你在决定是否使用 subagent 时，问自己一个问题：**中间过程的工作重要吗？**

如果答案是**不**——你只需要最终结果——把它委派给 subagent。如果答案是**是**——你需要看到并对沿途发生的事情做出反应——把它留在主线程中。

**适合使用 subagent 的场景：**

- 研究和探索
- Code review
- 需要自定义 system prompt 的任务

**避免使用 subagent 的场景：**

- 不增加真正能力的"专家"人设
- 每一步都依赖上一步的多步骤流水线
- 运行测试（你需要完整输出来调试）

> **[Claude 注释]** 让我把这个决策规则画成一个简单的决策树：
>
> ```
> 你有一个任务要完成
>   │
>   ├── 中间过程重要吗？
>   │     │
>   │     ├── 是 → 留在主线程
>   │     │     例：调试、测试、需要迭代反馈的开发
>   │     │
>   │     └── 否 → 进一步判断：
>   │           │
>   │           ├── 任务会消耗大量 context 吗？
>   │           │     ├── 是 → 非常适合 subagent (研究、探索)
>   │           │     └── 否 → 可能不值得 subagent 的开销
>   │           │
>   │           ├── 需要"全新视角"吗？
>   │           │     ├── 是 → 适合 subagent (code review)
>   │           │     └── 否 → 取决于其他因素
>   │           │
>   │           └── 需要不同的"人格"/指令吗？
>   │                 ├── 是 → 适合 subagent (文案、样式等)
>   │                 └── 否 → 可能不需要 subagent
> ```
>
> **最终总结**：Subagent 不是万能药。它们是一种工具，在正确的场景下能大幅提升效率，在错误的场景下反而增加复杂度和信息损失。学会判断"什么时候用"和"什么时候不用"，比学会"怎么创建"更重要。
