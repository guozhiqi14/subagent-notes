# 第一章：What are subagents? -- 什么是 Subagent？

> **课程章节**: 1/4 | **视频时长**: 2:49
> **视频链接**: https://www.youtube.com/watch?v=jKErNxuxPXg

---

Subagent 是 Claude Code 可以将任务委派给的专门助手。你可以把它们想象成专注的帮手：每个 subagent 在自己的 context window（对话上下文窗口）中运行，完成它的工作，然后将摘要返回给主线程。中间的步骤——所有的文件读取、搜索和工具调用——都被隔离在 subagent 内部，不会弄乱你的主对话。

> **[Claude 注释]** 用一个现实世界的类比来理解 subagent：想象你是一家公司的 CEO（主线程），你不需要亲自去调研市场数据、审查合同细节、检查代码质量。你把这些任务分别交给市场分析师、法务和 QA 工程师（subagent）。他们各自独立工作，最后只给你一份简洁的报告。你不需要知道他们翻了多少文档、打了多少电话——你只需要结论。
>
> 这就是 subagent 的核心思想：**委派 + 隔离 + 摘要返回**。

---

## 为什么 Subagent 很重要

每次你和 Claude Code 聊天，你都在往主 context window 里添加内容。每一次工具调用、每一次文件读取、每一次搜索结果都存储在那里。这个空间是有限的，一旦填满，Claude 就开始丢失对话早期部分的追踪。

> **[Claude 注释]** 这里说的"空间有限"指的是 **token 限制**。每个 LLM 对话都有一个 context window 大小（比如 200K tokens）。所有的输入和输出都要消耗 token。打个比方：context window 就像你的工作台桌面。桌面空间有限，如果你把每份参考资料、每张草稿纸都摊在上面，很快就没地方放新东西了，而且你也找不到之前放的重要资料。
>
> **真实场景**：假设你让 Claude 读取一个项目的 15 个文件来回答一个问题。每个文件平均 500 行，那就是 7500 行代码被塞进了 context window。但你真正需要的答案可能只有一两句话。这 7500 行代码白白占据了你宝贵的"桌面空间"。

Subagent 通过启动一个**独立的 context window** 来解决这个问题。Subagent 收到两样东西：

- **一个自定义 system prompt**——来自你的配置文件，定义了 subagent 的角色和行为
- **一个任务描述**——由父 agent 根据你的请求编写

然后 subagent 就开始独立工作。它读取文件、运行搜索、编辑代码——做任何它需要做的事。完成后，只有一个**摘要**返回到你的主对话。整个 subagent 的对话随后被丢弃。

这意味着你的主 context 保持干净。你得到了答案，却没有寻找答案过程中的所有噪音。代价是你失去了对 subagent 如何得出结论的可见性。

> **[Claude 注释]** 这个 tradeoff（权衡）非常关键。用一个更直观的例子：
>
> - **不用 subagent**：你让实习生在你旁边的工位上工作，你能看到他的每一步操作。好处是完全透明，坏处是他的所有草稿和搜索结果都堆在你的桌上。
> - **用 subagent**：你让实习生回自己工位工作，完成后给你交一份报告。好处是你的桌面干净，坏处是如果报告有问题，你不知道他中间经历了什么。
>
> **底线**：当你只关心结果、不关心过程时，用 subagent。当你需要看到中间步骤来做判断时，不要用。

## 一个实际的例子

假设你在探索一个不熟悉的代码库，你想知道哪个服务处理退款（refund）。如果不用 subagent，Claude 可能会读取 15 个文件、运行多次搜索、追踪多个函数调用。所有这些都会填满你的 context window，即使你只需要一个事实。

用 subagent 的体验就干净多了。你提出问题，Explore subagent 启动，在它自己的 context 中完成所有的挖掘工作，然后返回一个专注的答案。

你的主 context window 只记录了问题和摘要——而不是沿途读取的 15 个文件。

> **[Claude 注释]** 这里的数字很说明问题。假设每个文件平均消耗 2000 tokens：
>
> | 方式 | Context window 消耗 |
> |------|-------------------|
> | 不用 subagent | 问题(~50 tokens) + 15个文件(~30,000 tokens) + 答案(~200 tokens) = **~30,250 tokens** |
> | 用 subagent | 问题(~50 tokens) + 摘要返回(~500 tokens) = **~550 tokens** |
>
> 差了约 **55 倍**！这就是为什么 subagent 对长会话如此重要——它们让你的 context window "续航"更久。

## 内置 Subagent

Claude Code 自带几个内置 subagent，你可以直接使用：

- **General purpose subagent（通用型）**——用于需要探索和操作的多步骤任务
- **Explore**——用于快速搜索和浏览代码库
- **Plan**——在 plan mode 中用于研究和分析你的代码库，然后再呈现一个方案

> **[Claude 注释]** 这三个内置 subagent 覆盖了最常见的使用场景：
>
> | Subagent | 模型 | 工具权限 | 典型场景 |
> |----------|------|---------|---------|
> | **Explore** | Haiku（快速、低延迟） | 只读（不能编辑文件） | "这个 API 在哪个文件里定义的？" |
> | **Plan** | 继承主会话模型 | 只读 | Plan mode 下的代码库调研 |
> | **General purpose** | 继承主会话模型 | 所有工具 | 复杂的多步骤任务 |
>
> 注意 **Explore 用的是 Haiku 模型**——这是一个有意的设计选择。Explore 的工作是快速搜索和浏览，不需要深度推理，所以用更快、更便宜的模型就够了。这也体现了 subagent 的另一个优势：**你可以为不同任务选择不同的模型，从而优化成本和速度**。

## 自定义 Subagent

除了内置选项之外，你还可以用自定义 system prompt 和工具访问权限创建自己的 subagent。这让你可以定义针对你工作流的专门 agent——一个 code reviewer、一个测试编写器、一个文档生成器，或者任何你需要的东西。

> **[Claude 注释]** 自定义 subagent 是这门课程后续章节的重点。简单来说，它就是一个 Markdown 文件，放在 `.claude/agents/` 目录下，用 YAML frontmatter 定义元数据，用 Markdown 正文作为 system prompt。后面的章节会详细展开。

## 核心要点

Subagent 给你三个主要好处：

1. **它们将工作拆解为专注的小块**，让每个 subagent 专注于一个特定任务
2. **它们保持主 context window 的干净**，通过隔离所有的中间工作
3. **它们只带回你需要的信息**，以简洁的摘要形式

无论你是使用内置 subagent 还是创建自己的，它们都是在更长的 Claude Code 会话中获得更多产出的实用方式。主 context 中的噪音越少，你就能越长久、越高效地工作。

> **[Claude 注释]** 从第一性原理来总结这一章：
>
> **问题**：LLM 的 context window 是有限资源，长会话中很容易被中间过程消耗殆尽。
>
> **解决方案**：Subagent = 独立 context window + 任务隔离 + 摘要返回。
>
> **本质**：这其实就是计算机科学中的 **进程隔离** 思想。主进程（主线程）不需要知道子进程（subagent）的所有内部状态，只需要通过一个定义好的接口（摘要）获取结果。这种模式在操作系统、微服务架构中随处可见。
>
> 唯一的区别是：传统的进程隔离是为了安全和稳定性，而 subagent 的隔离主要是为了**节省 context window 空间**——一种 LLM 时代特有的资源约束。
